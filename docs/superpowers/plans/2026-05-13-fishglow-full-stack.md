# 渔光 FishGlow — 产品深度调研与开发规划

---

## 一、产品调研分析

### 1.1 产品定位

| 维度 | 分析 |
|------|------|
| **产品类型** | 工具型 + 收集型小程序（钓鱼垂直赛道） |
| **核心价值** | 拍照识鱼（AI能力）→ 百科学习 → 宝可梦式图鉴收集 |
| **目标用户** | 钓鱼爱好者（新手到进阶），年龄25-55岁，男性为主 |
| **使用场景** | 户外钓鱼现场、钓后复盘分享、日常学习鱼种知识 |
| **商业模式** | MVP阶段免费，后期可探索：会员(高级识别)、社区、钓具推荐 |
| **竞品分析** | 识鱼类APP较少且体验差，此产品聚焦微信生态+游戏化收集是差异点 |

### 1.3 技术可行性评估

| 技术点 | 成熟度 | 风险 | 备注 |
|--------|--------|------|------|
| 百度动物识别API | 成熟 | 低 | 识别准确率依赖图片质量，鱼类子类识别可能不够精细 |
| Mimo大模型结构化 | 可用 | 中 | 超时风险15s，需做好降级（structured=null时展示原文） |
| 微信小程序相机 | 成熟 | 低 | 需注意权限获取、全屏适配 |
| uni-app编译小程序 | 成熟 | 低 | Vue 3 + Vite 方案稳定 |
| SQLite on 服务器 | 轻量 | 低 | 单机足够支撑，需WAL模式防并发写入 |
| 腾讯云轻量 | 稳定 | 低 | 已配好HTTPS+域名+Nginx |

### 1.4 数据资产盘点

#### 鱼种季节数据 (fish_seasonal.json)

- **覆盖**: 30种淡水鱼，涵盖中国主要垂钓鱼种
- **难度分布**: easy(10种) / medium(14种) / hard(6种) — 合理分布
- **钓法覆盖**: 24种细分钓法（台钓/路亚/传统钓/抛竿/飞蝇钓等）
- **数据质量**: 每种鱼含12月活跃度评分(1-10) + 4季饵料 + 4季时间 + 水温范围 + 栖息地 + tips
- **当月(5月)**: 7种鱼活跃度达10分，适合作为推荐首选

#### 美术素材

| 素材 | 状态 | 质量评估 |
|------|------|----------|
| 勋章PNG ×8 | ✅ 已有 | 732KB-1.1MB，需压缩优化用于移动端 |
| banner.jpg | ✅ 已有 | 78KB，尺寸合适 |
| SVG图标 ×15 | ✅ 已有 | 195B-1.9KB，矢量格式，需生成三色变体 |

#### SVG 图标格式分析

```
当前颜色: stroke="#0284C7" (天蓝色)
需要生成:
  - primary: #EA580C (橙色) → 选中态/强调
  - muted: #94A3B8 (灰色) → 默认态
  - white: #FFFFFF → 深色背景上使用
```

图标采用 stroke 描边风格（非 fill 填充），统一 stroke-width="4"，视觉一致性好。

│                  │         │  │  ├─ Services       │  │

### 2.2 API 接口清单 (14个)

| # | 方法 | 路径 | 核心职责 | 复杂度 |
|---|------|------|----------|--------|
| 1 | POST | /api/auth/openid | 微信登录换openid | 低 |
| 2 | POST | /api/recognize | 上传图片→AI识别→结构化 | **高** |
| 3 | POST | /api/structurize | 按需结构化百科文本 | 中 |
| 4 | GET | /api/seasonal | 当月鱼种推荐 | 低 |
| 5 | GET | /api/badges | 勋章图片URL映射 | 低 |
| 6 | GET | /api/user | 获取用户信息 | 低 |
| 7 | POST | /api/user/update | 更新昵称/头像 | 低 |
| 8 | POST | /api/collection/add | 收藏鱼种(含图片上传) | 中 |
| 9 | GET | /api/collection/list | 图鉴列表 | 低 |
| 10 | POST | /api/history/add | 添加识别历史 | 中 |
| 11 | GET | /api/history/list | 历史列表(分页) | 低 |
| 12 | GET | /api/achievements | 用户成就列表 | 低 |
| 13 | GET | /api/user/stats | 统计数据 | 低 |
| 14 | GET | /api/health | 健康检查 | 低 |

**关键链路**: `拍照 → /api/recognize → 百度AI → Mimo结构化 → 返回` (耗时预估3-8s)

### 2.3 数据库设计分析

```
users (用户表) ── 1:N ──┬── collections (图鉴表, UNIQUE: openid+fish_name)
                         ├── history (识别历史, 允许重复)
                         └── achievements (成就表, UNIQUE: openid+badge_id)
```

**设计合理性**:
- collections 按 fish_name 去重 → 符合"图鉴收集"语义
- history 不去重 → 保留每次识别记录
- achievements 去重 → 勋章只解锁一次
- 无外键约束 → SQLite 适合轻量场景

### 2.4 页面结构分析

```
TabBar (5页)
├── 首页 /home ─────── 发现Feed (聚合: 每日一鱼+挑战+当季+入口)
├── 鱼库 /fishdb ───── 搜索+分类筛选 (数据源: fish_seasonal.json)
├── 识鱼 /identify ─── 全屏相机 (核心交互入口)
├── 图鉴 /collection ─ 3列网格收集册
└── 我的 /profile ──── 用户中心

二级页 (4页)
├── /result ─────── 识别结果 (最复杂: 候选列表+结构化+收藏)
├── /fish-detail ── 鱼种详情 (月历+钓法+饵料)
├── /achievement ── 成就墙 (2列勋章)
└── /history ────── 识别历史 (时间线列表)
```

## 三、风险识别与应对

| 风险 | 等级 | 影响 | 应对策略 |
|------|------|------|----------|
| 百度AI对鱼类识别精度不足 | 中 | 用户体验 | 展示Top6结果+置信度标签让用户选择 |
| Mimo超时(>15s) | 中 | 结果页卡顿 | 超时返回structured=null，前端展示原始百科描述 |
| 百度token过期 | 低 | 识别失败 | 内存缓存+提前1天刷新+并发锁 |
| 勋章图片过大(~1MB) | 中 | 加载慢 | 服务端用Sharp压缩到200KB内，或提供缩略图 |
| 小程序审核 | 中 | 上线延迟 | 确保分类正确(工具类)，相机权限说明清晰 |
| 单机SQLite并发写 | 低 | 数据丢失 | WAL模式 + 重试机制 |

## 四、开发规划

### 4.1 阶段划分与工时预估

| 阶段 | 内容 | 预估 | 依赖 | 可并行 |
|------|------|------|------|--------|
| **P1: 后端核心** | Express + DB + 百度AI + Mimo + 14个API | 优先 | 无 | ✅ |
| **P2: 前端基础** | uni-app初始化 + 路由 + tabBar + 工具层 + 图标 | 优先 | 无 | ✅ (与P1并行) |
| **P3: 核心页面** | 识鱼页 + 结果页 + 首页 | 次优先 | P1+P2 | 部分并行 |
| **P4: 功能页面** | 鱼库 + 详情 + 图鉴 + 我的 + 历史 + 成就 | 中 | P3 | 页面间可并行 |
| **P5: 完善** | 分享功能 + 错误处理 + 性能优化 + 图片压缩 | 后 | P4 | — |
| **P6: 部署上线** | 服务器部署 + 小程序审核提交 | 最后 | P5 | — |

### 4.2 并行开发策略

```
时间线 ──────────────────────────────────────────→

Agent A (后端):  [P1: Express+DB+Services+Routes]──────→ [联调]
Agent B (前端):  [P2: 初始化+工具层]→[P3: 识鱼+结果+首页]→[P4: 其他页面]→[P5]
Agent C (设计):  [图标三色化]→[UI组件设计] (使用 ui-ux-pro-max)

并行点:
  • P1 和 P2 完全独立，可同时开发
  • P3 中识鱼页的 UI 可先 mock 数据开发，待 P1 完成后联调
  • P4 中各页面相互独立，可分配多个子Agent
```

### 4.4 设计开发注意事项

#### 设计规范（开发时使用 `/ui-ux-pro-max` skill）

- **风格**: 浅色底 + 高饱和度橙(#EA580C)主色 + 日系卡片风格
- **卡片**: 直角矩形(Fluent Design)，border-radius: 2-4rpx
- **图标**: IconPark SVG stroke风格，不用emoji
- **间距**: 8rpx倍数 (16/24/32/48)
- **字体**: 系统字体，28rpx基准
- **动效**: 按压opacity:0.7，过渡150-200ms

#### 关键踩坑点（开发前必读）

1. `uploadFile` 字段名必须是 `file`
2. `structured` 是对象 `{alias, family, ...}` 不是数组
3. mimo 必须 `thinking: { type: "disabled" }`
4. 季节API返回 `res.data.data.recommendations`
5. 鱼库筛选字段是 `fishing_methods`（数组）
6. 相机页: `page { height: 100vh; background: #000 }`
7. PM2: `--cwd /opt/fishglow/server`
8. tabBar页用 `uni.switchTab`，非tabBar用 `navigateTo`
9. 百度token: 内存缓存+并发锁
10. 小程序不支持 @font-face，图标用 SVG `<image>`
